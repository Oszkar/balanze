# v0.5.1 Reach (CLI distribution) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship prebuilt `balanze-cli` binaries for four targets plus a Homebrew tap, so the CLI installs on Windows, macOS, and Linux without a Rust toolchain.

**Architecture:** Almost all of this is CI and packaging. `.github/workflows/release.yml` gains a `notes` job (single source of release-body prose), a `cli` matrix job (four targets, archives plus `.sha256` siblings), and a new `.github/workflows/tap.yml` fires on `release: published` to rewrite a Homebrew formula and cask in a separate tap repository. The single Rust change adds a typed `KeychainError::NoStore` so Linux, which has no credential store wired, gives `BALANZE_OPENAI_KEY` guidance instead of a raw keyring error.

**Tech Stack:** GitHub Actions, Rust 1.94 (edition 2024), `cargo-deny`, Homebrew (Ruby formula + cask), `gh` CLI.

**Spec:** `docs/superpowers/specs/2026-07-21-v0.5.1-reach-design.md`. Read it before starting - it records why each decision was made.

## Task 2: Typed no-store keychain error

`crates/keychain` registers no credential store on Linux: `init_default_store` logs a warning under `cfg(not(any(windows, target_os = "macos")))` and leaves keyring-core storeless. Every `get` / `set` / `delete` then returns `KeychainError::PlatformError` carrying raw keyring-core text.

The read path is already fine - `resolve_openai_key` gives `BALANZE_OPENAI_KEY` precedence, so a Linux user with the env var set never touches the keychain. The problem is the **write** path and the wizard, which surface the raw error.

This adds a typed variant so consumers can distinguish "this platform has no store" (expected, documented) from "the store is broken" (a real fault). AGENTS.md §3.2 calls for exactly this when a consumer pattern-matches variants.

**Files:**
- Modify: `crates/keychain/src/lib.rs:59-66` (the error enum), `:69-82` (`set`), `:88-103` (`get`), `:144-162` (`delete`)
- Modify: `crates/balanze_cli/src/keys.rs:51`, `:69`
- Modify: `crates/balanze_cli/src/setup.rs:47-53` (the status enum), `:59-68` (`summary_line`), `:189-193` (the first keychain read)
- Modify: `crates/balanze_cli/src/probes.rs:341-352` (`KeychainHealth`), `:377-407`, `:460-470`, `:489-506`
- Test: inline `#[cfg(test)] mod tests` in `crates/keychain/src/lib.rs` and `crates/balanze_cli/src/probes.rs`

**Interfaces:**
- Consumes: nothing from Task 1.
- Produces:
  - `keychain::KeychainError::NoStore` - a unit variant, `Display` text `"no OS credential store is available on this platform"`.
  - `keychain::has_native_store() -> bool` - `true` on Windows and macOS, `false` elsewhere. Pure, const-evaluable, testable on every platform.
  - `keychain::NO_STORE_HINT: &str` - the shared guidance string, so the CLI does not restate it in four places.
  - `keychain::resolve_openai_key()` keeps its signature but changes behavior: `NoStore` now resolves to `Ok(None)` ("not configured") rather than propagating. Every consumer of that function inherits the change.
  - `balanze_cli::probes::KeychainHealth::NotAvailable` - a new variant.
  - `balanze_cli::setup::OpenAiKeyStatus::NoKeychainOnPlatform` - a new variant (private to the module).

In `crates/keychain/src/lib.rs`, replace the `KeychainError` enum (lines 59 to 66) with:

```rust
#[derive(Debug, Error)]
pub enum KeychainError {
    #[error("no entry found for `{0}`")]
    NotFound(String),

    #[error("OS keychain access failed for `{name}`: {reason}")]
    PlatformError { name: String, reason: String },

    /// This platform has no native credential store wired (see
    /// [`init_default_store`]). Distinct from [`KeychainError::PlatformError`]
    /// because it is an expected, documented platform condition rather than a
    /// fault: callers should route the user to [`NO_STORE_HINT`] instead of
    /// reporting a broken keychain.
    #[error("no OS credential store is available on this platform")]
    NoStore,
}

/// Guidance for the [`KeychainError::NoStore`] case. Kept here so every surface
/// that hits a storeless platform tells the user the same thing.
pub const NO_STORE_HINT: &str =
    "Set the BALANZE_OPENAI_KEY environment variable instead; it takes precedence over the keychain.";

/// Whether this platform has a native credential store wired.
///
/// Mirrors the `cfg` on the store registration in [`init_default_store`]. Adding
/// a backend for a new platform means updating both.
pub const fn has_native_store() -> bool {
    cfg!(any(windows, target_os = "macos"))
}
```

`resolve_openai_key_with` must **not** propagate it. Its contract is a configured/not question, and its catch-all would turn a storeless platform into a hard error for every consumer. Replace the function body (currently at line 126) with:

```rust
fn resolve_openai_key_with(env_var: Option<String>) -> Result<Option<String>, KeychainError> {
    // Present (blank or not) is authoritative; a blank value is `None` here and
    // must not fall through to the keychain below.
    if let Some(raw) = env_var {
        return Ok(non_blank(&raw));
    }
    match get(keys::OPENAI_API_KEY) {
        Ok(k) => Ok(Some(k)),
        Err(KeychainError::NotFound(_)) => Ok(None),
        // No store means nothing can ever have been stored, so "not configured"
        // is literally true. The typed error stays on get/set/delete, where the
        // CLI needs it to tell the user where to put the key instead.
        Err(KeychainError::NoStore) => Ok(None),
        Err(e) => Err(e),
    }
}
```

Why this matters, concretely. Without it:

- `crates/balanze_cli/src/sources.rs:280` (`live_fetch_openai`) propagates with `?`, so an unconfigured Linux user gets an **error** in the OpenAI cell instead of a neutral not-configured state. Its own doc comment promises "`Err` only for real fetch failures (401, 403, network, etc.)". AGENTS.md §3.4 draws exactly this line between an error slot and a neutral absence.
- `crates/watcher/src/tasks/openai_poll.rs:117` swallows the error but logs `warn!("keychain error (treating as not configured)")`, so every Linux launch emits a warning about a permanent platform condition.

Add a matching sentence to `resolve_openai_key`'s doc comment (line 109), which currently reads "`Err` is a real keychain failure (locked, denied), not mere absence":

```rust
/// `Ok(None)` means "not configured", including on a platform with no credential
/// store at all. `Err` is a real keychain failure (locked, denied), not mere
/// absence.
```

- [ ] **Step 4: Add the `cli` job**

Append this job to `.github/workflows/release.yml`, after `build`:

```yaml
  cli:
    # Prebuilt `balanze-cli` binaries. `needs: build` because the desktop legs
    # are what create the draft release; `gh release upload` needs it to exist.
    # This matrix plus `build`'s matrix together define the shipped
    # architectures, which deny.toml's [graph].targets must match as a union
    # (AGENTS.md §1).
    needs: build
    name: cli (${{ matrix.target }})
    strategy:
      fail-fast: false
      matrix:
        include:
          - platform: ubuntu-latest
            target: x86_64-unknown-linux-musl
            archive: tar.gz
          - platform: macos-latest
            target: aarch64-apple-darwin
            archive: tar.gz
          - platform: windows-latest
            target: x86_64-pc-windows-msvc
            archive: zip
          - platform: windows-latest
            target: aarch64-pc-windows-msvc
            archive: zip

    runs-on: ${{ matrix.platform }}

    steps:
      - name: Checkout
        uses: actions/checkout@v7
        with:
          ref: ${{ inputs.tag }}

      - name: Install musl toolchain
        # ring is the CLI's only non-Rust dependency (reqwest runs on rustls, no
        # OpenSSL), and it needs a C compiler targeting musl. Static musl is
        # deliberate: a glibc binary built here would link this runner's glibc
        # and refuse to run on older distros.
        if: matrix.target == 'x86_64-unknown-linux-musl'
        run: sudo apt-get update && sudo apt-get install -y musl-tools

      - name: Install Rust toolchain
        # Pin (channel + components) comes from rust-toolchain.toml - see ci.yml.
        uses: actions-rust-lang/setup-rust-toolchain@v1
        with:
          target: ${{ matrix.target }}
          cache: false
          rustflags: ""

      - name: Cache Rust workspace
        uses: swatinem/rust-cache@v2
        with:
          key: cli-${{ matrix.target }}

      - name: Build balanze-cli
        run: cargo build --release --locked --package balanze_cli --target ${{ matrix.target }}

      - name: Package, checksum, and upload
        shell: bash
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          TAG: ${{ inputs.tag }}
          TARGET: ${{ matrix.target }}
          ARCHIVE: ${{ matrix.archive }}
        run: |
          set -euo pipefail
          NAME="balanze-cli-${TAG}-${TARGET}"
          mkdir -p "dist/${NAME}"
          if [ "$ARCHIVE" = "zip" ]; then
            cp "target/${TARGET}/release/balanze-cli.exe" "dist/${NAME}/"
          else
            cp "target/${TARGET}/release/balanze-cli" "dist/${NAME}/"
          fi
          cp LICENSE "dist/${NAME}/"

          cd dist
          if [ "$ARCHIVE" = "zip" ]; then
            ASSET="${NAME}.zip"
            7z a -tzip "$ASSET" "$NAME" > /dev/null
          else
            ASSET="${NAME}.tar.gz"
            tar -czf "$ASSET" "$NAME"
          fi

          # macOS runners ship BSD shasum, not coreutils' sha256sum; Git Bash on
          # the Windows runner ships sha256sum. Same idiom as the desktop legs'
          # checksum step.
          if command -v sha256sum > /dev/null 2>&1; then
            HASH=(sha256sum)
          else
            HASH=(shasum -a 256)
          fi
          "${HASH[@]}" "$ASSET" > "${ASSET}.sha256"
          cat "${ASSET}.sha256"

          # One sibling .sha256 per archive rather than a shared checksum file:
          # the four legs race, and `--clobber` on a shared name would drop
          # whichever finished first. Homebrew also wants exactly one hash per
          # URL, which this gives it directly.
          gh release upload "$TAG" "$ASSET" "${ASSET}.sha256" --clobber
```

- [ ] **Step 6: Create the tap workflow**

Create `.github/workflows/tap.yml`:

```yaml
name: tap

# Fires when a release is PUBLISHED, never during the build. release.yml drafts
# the release, and a draft's asset URLs do not resolve for anonymous users - a
# formula written during the build would point at a URL that 404s until the
# maintainer clicks Publish, and Homebrew users are anonymous. Publication is
# also the correct semantic trigger: it is exactly when the URLs go live.
#
# workflow_dispatch is NOT redundant. A `release` event runs this file as it
# exists at the release's tag ref, so re-running a failed run replays the same
# broken file, and a fix merged to main is never picked up - there is no second
# `published` event for an already-published release. The dispatch path gives
# replay-after-fix, a validation route against an older tag, and recovery from
# an unrelated failure (expired token, tap-repo conflict). The update is
# idempotent, so re-running against an already-current tag is a no-op.
on:
  release:
    types: [published]
  workflow_dispatch:
    inputs:
      tag:
        description: 'Published release tag to sync the tap to (e.g. v0.5.1)'
        required: true
        type: string

permissions:
  contents: read

jobs:
  update:
    runs-on: ubuntu-latest
    env:
      # `github.event.release.tag_name` exists only on the release trigger.
      TAG: ${{ github.event.release.tag_name || inputs.tag }}
    steps:
      - name: Validate the tag and confirm the release is published
        # On the dispatch path the tag is user-supplied and the release may
        # still be a draft, whose asset URLs 404 for anonymous users - exactly
        # the state this workflow exists to avoid writing into the tap.
        shell: bash
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          set -euo pipefail
          if [[ ! "$TAG" =~ ^v[0-9]+\.[0-9]+\.[0-9]+$ ]]; then
            echo "::error::tag '$TAG' must match v*.*.* (e.g. v0.5.1)"
            exit 1
          fi
          is_draft=$(gh release view "$TAG" --repo "${{ github.repository }}" --json isDraft --jq .isDraft)
          if [ "$is_draft" != "false" ]; then
            echo "::error::release $TAG is still a draft; its asset URLs do not resolve for anonymous users"
            exit 1
          fi

      - name: Check out the tap
        # GITHUB_TOKEN cannot push to another repository, so this uses a
        # fine-grained PAT scoped to `contents: write` on the tap repo alone.
        uses: actions/checkout@v7
        with:
          repository: Oszkar/homebrew-balanze
          token: ${{ secrets.TAP_PUSH_TOKEN }}
          path: tap

      - name: Rewrite the formula and cask
        shell: bash
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          set -euo pipefail
          VERSION="${TAG#v}"

          # Read the checksums the release itself advertises rather than
          # re-hashing, so the formula can only ever agree with the release.
          mkdir -p sums
          gh release download "$TAG" --repo "${{ github.repository }}" \
            --pattern '*.sha256' --dir sums
          gh release download "$TAG" --repo "${{ github.repository }}" \
            --pattern 'macos-aarch64-checksums.txt' --dir sums

          mac_sha=$(cut -d' ' -f1 "sums/balanze-cli-${TAG}-aarch64-apple-darwin.tar.gz.sha256")
          linux_sha=$(cut -d' ' -f1 "sums/balanze-cli-${TAG}-x86_64-unknown-linux-musl.tar.gz.sha256")
          dmg_name="Balanze_${VERSION}_aarch64.dmg"
          dmg_sha=$(awk -v n="$dmg_name" '$2 == n { print $1 }' sums/macos-aarch64-checksums.txt)

          for v in mac_sha linux_sha dmg_sha; do
            if [ -z "${!v}" ]; then
              echo "::error::could not resolve $v from the release checksums"
              exit 1
            fi
          done

          base="https://github.com/${{ github.repository }}/releases/download/${TAG}"

          cat > tap/Formula/balanze-cli.rb <<EOF
          class BalanzeCli < Formula
            desc "Consolidated personal AI usage tracking for Claude and Codex"
            homepage "https://github.com/${{ github.repository }}"
            version "${VERSION}"
            license "MIT"

            on_macos do
              on_arm do
                url "${base}/balanze-cli-${TAG}-aarch64-apple-darwin.tar.gz"
                sha256 "${mac_sha}"
              end
            end

            on_linux do
              on_intel do
                url "${base}/balanze-cli-${TAG}-x86_64-unknown-linux-musl.tar.gz"
                sha256 "${linux_sha}"
              end
            end

            def install
              bin.install "balanze-cli"
            end

            test do
              assert_match version.to_s, shell_output("#{bin}/balanze-cli --version")
            end
          end
          EOF

          cat > tap/Casks/balanze.rb <<EOF
          cask "balanze" do
            version "${VERSION}"
            sha256 "${dmg_sha}"

            url "https://github.com/${{ github.repository }}/releases/download/v#{version}/Balanze_#{version}_aarch64.dmg"
            name "Balanze"
            desc "Tray app for consolidated personal AI usage tracking"
            homepage "https://github.com/${{ github.repository }}"

            depends_on macos: ">= :sequoia"
            depends_on arch: :arm64

            app "Balanze.app"

            zap trash: [
              "~/Library/Application Support/me.oszkar.balanze",
              "~/Library/Caches/me.oszkar.balanze",
              "~/Library/Preferences/me.oszkar.balanze.plist",
              "~/Library/Saved Application State/me.oszkar.balanze.savedState",
            ]
          end
          EOF

      - name: Commit and push the tap
        working-directory: tap
        run: |
          set -euo pipefail
          git config user.name "github-actions[bot]"
          git config user.email "41898282+github-actions[bot]@users.noreply.github.com"
          # Idempotency: a dispatch replay against an already-current tag is a
          # no-op, not a failure. This is what makes the dispatch path safe to
          # re-run after fixing an unrelated step.
          if git diff --quiet; then
            echo "tap already up to date for $TAG"
            exit 0
          fi
          git add Formula/balanze-cli.rb Casks/balanze.rb
          git commit -m "chore: update balanze to ${TAG}"
          git push
```

**Critical detail:** the two heredocs are **unquoted** (`<<EOF`, not `<<'EOF'`) so shell variables expand. That means `#{bin}` and `#{version}` inside the Ruby survive only because `#{...}` is not shell syntax - but `$` would be eaten. There are no `$` characters in the emitted Ruby, which is why this is safe. Do not add any.

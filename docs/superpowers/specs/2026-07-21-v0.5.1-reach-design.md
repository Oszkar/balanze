# v0.5.1 - Reach (CLI distribution) - Milestone Spec

Date: 2026-07-21
Status: approved (scope), ready for implementation plan
Phase: 5 (Distribution & Legibility), second sub-milestone

## 1. Context

v0.5.0 shipped the first binary release: a signed and notarized macOS DMG/app plus unsigned Windows MSI/NSIS, launch-at-login, and a README refresh. That covered the **desktop** app only.

The CLI is still source-only. `cargo install --git` remains the sole install path for `balanze-cli`, which is also the entire Linux story (`docs/PRD.md` "Supported platforms" lists Linux as CLI-only). `.github/workflows/release.yml` has no Linux leg and builds no CLI binary at all - `tauri-action` produces desktop bundles and nothing else.

v0.5.1 closes that gap. Theme: **you can install Balanze's CLI on any of the three OSes without a Rust toolchain.**

Phase 5's remaining sub-milestones, decided alongside this spec:

- **v0.5.1 - Reach** (this spec): CLI distribution + Windows arm64 for the CLI.
- **v0.5.2 - Legibility**: the mdBook docs site (#210) and the "how it works" writeup.
- **v0.5.3 - Support & hardening**: the price-table refresh script, the "Send Logs" menu item, and the accumulated correctness debt (#214, #198, #204, and the #107 / #152 pace call).

Then v0.6 (Alerts).

## 2. Scope

### In scope

1. **Prebuilt `balanze-cli` binaries** for four targets, attached to the GitHub Release with checksums.
2. **Windows arm64, CLI only** - the cheap half of #219. `balanze_cli` has no WebView2 dependency, so the desktop half of that issue stays open with its runner and bundler questions intact.
3. **A Homebrew tap** (`Oszkar/homebrew-balanze`) carrying a `balanze-cli` formula and a `balanze` desktop cask.
4. **A typed no-store keychain error** so Linux users get actionable guidance instead of a raw `PlatformError`.
5. **Release-notes de-duplication** in `release.yml`.
6. **Changelog and roadmap housekeeping** (see 4.1).

### Deferred to v0.5.2+

- The mdBook docs site (#210) and the "how it works" writeup - v0.5.2.
- Price-table refresh script, "Send Logs", #214, #198, #204, the #107 / #152 pace call - v0.5.3.
- Desktop arm64 Windows - the rest of #219, unscheduled.
- A native Linux credential store - filed as a follow-up, not built here (see 3.2).
- #114 (CLI export filters), #134, #186, #85 - unscheduled.

### Done condition

- Three CLI archives plus checksums are attached to the published `v0.5.1` release: Linux musl, macOS arm64, Windows x64. **Windows arm64 is a fourth, conditional target** - see below.
- `brew install oszkar/balanze/balanze-cli` works on macOS (Apple Silicon) and on Linuxbrew (x64), each verified by a clean install on that OS.
- `brew install --cask oszkar/balanze/balanze` installs the notarized desktop app.
- `balanze-cli keys set` on Linux fails with `BALANZE_OPENAI_KEY` guidance, not a raw keyring error.
- `docs/PRD.md`'s platform matrix, `release.yml`'s build matrices, and `deny.toml`'s `[graph].targets` agree (AGENTS.md §1) - whichever set of targets actually ships.
- `CHANGELOG.md` has a `## [0.5.1]` section, and the retroactively-missing `## [0.5.0]` one.

**Windows arm64 is explicitly conditional, decided now rather than mid-flight.** It ships if `aarch64-pc-windows-msvc` builds on the stock `windows-latest` image. If it does not - most likely because the ARM64 MSVC build tools are absent and `ring` cannot compile - the target is dropped from all three lists in the same commit, #219 is updated, and the milestone still completes. Rationale: #219 argues on its own merits that nobody is blocked, because x64 binaries run under emulation on arm64 Windows. Blocking a distribution milestone on a runner-image question would be the wrong trade. What is **not** acceptable is dropping it silently: the three lists must still agree afterwards, and the README install table must not offer a file that was never built.

Installing ARM64 toolchains into CI to force the leg through is out of scope. That is a separate decision belonging to #219.

## 3. Settled decisions

### 3.1 Targets and artifact naming

| Target | Runner | Archive |
|---|---|---|
| `x86_64-unknown-linux-musl` | `ubuntu-latest` + `musl-tools` | `balanze-cli-v0.5.1-x86_64-unknown-linux-musl.tar.gz` |
| `aarch64-apple-darwin` | `macos-latest` | `balanze-cli-v0.5.1-aarch64-apple-darwin.tar.gz` |
| `x86_64-pc-windows-msvc` | `windows-latest` | `balanze-cli-v0.5.1-x86_64-pc-windows-msvc.zip` |
| `aarch64-pc-windows-msvc` | `windows-latest` (cross-compiled) | `balanze-cli-v0.5.1-aarch64-pc-windows-msvc.zip` |

Each archive contains the binary plus `LICENSE`, and nothing else.

Cross-compilation is cheap here because of the workspace's zero-syslib promise: `reqwest` is `default-features = false` with `rustls-tls`, so `ring 0.17` is the only non-Rust dependency in the CLI tree. It supports every target above. The arm64 Windows leg needs the ARM64 MSVC build tools, which the `windows-latest` image carries; if that turns out to be false, the leg is dropped rather than worked around, and #219 keeps the whole arm64 question.

**Linux uses static musl, not glibc.** A `x86_64-unknown-linux-gnu` binary built on `ubuntu-latest` links glibc 2.39 and will not run on Debian 12 or Ubuntu 22.04. Static musl runs on any Linux regardless of distro age, costs one `apt install musl-tools` step, and matches the zero-syslib promise: one file, no runtime dependencies.

**Checksums are per-archive `.sha256` siblings**, not the desktop legs' `<leg>-checksums.txt` shape. The four CLI legs race the same way the desktop legs do (which is why `release.yml:180` already writes one checksum file per leg), and Homebrew needs exactly one sha256 value per URL. A sibling file satisfies both without an aggregation step.

### 3.2 Linux secret storage - env var only, documented

`crates/keychain` registers no store on Linux. `init_default_store` (`crates/keychain/src/lib.rs:31`) logs a warning under `cfg(not(any(windows, target_os = "macos")))` and leaves keyring-core storeless, so every get/set/delete returns `KeychainError::PlatformError`.

v0.5.1 does **not** wire a Secret Service backend. Doing so would add a DBus dependency to the one crate allowed to hold keyring dependencies, require a headless-server degraded path, and turn a packaging release into a secrets-boundary release (AGENTS.md §3.4, §8). It is filed as a follow-up issue instead.

Adding a typed variant touches more than the write path. The full site list:

| Site | Today | After |
|---|---|---|
| `keychain::resolve_openai_key_with` (`lib.rs:132`) | `Err(e) => Err(e)` propagates | **Maps `NoStore` to `Ok(None)`.** With no store there is no stored key, so "not configured" is literally true. Without this, `sources.rs:280`'s `live_fetch_openai` returns `Err` for an unconfigured Linux user, violating its own documented contract ("`Err` only for real fetch failures"), and `openai_poll.rs:117` logs a `warn!` on every Linux launch for a permanent platform condition. AGENTS.md §3.4 calls for the neutral not-configured path here, not an error slot. |
| `probes.rs:377` presence probe | Exhaustive match, no `NoStore` arm | **Will not compile without a new arm.** Reports a `warn` naming the env override, and returns the new `KeychainHealth::NotAvailable`. |
| `probes.rs:463` `resolve_keychain_health` | Exhaustive match | **Will not compile without a new arm.** Maps to `NotAvailable`. |
| `probes.rs:494` `probe_settings_and_keychain` | Reports `Broken` as a failed check | `NotAvailable` becomes a **warning**, not a failure. A storeless platform is the documented Linux state, not a fault. |
| `setup.rs:189` the wizard's first `get` | `Err(e) => return Err(e.into())` | **Fires before any write site is reached**, so mapping only the writes would leave `setup` failing with the raw error. Bails out of the OpenAI step before prompting, since prompting for an unstorable key wastes the user's time. |
| `keys.rs:51` `set-openai-key` | Bare `?` | Explains there is nowhere to store the key. |
| `keys.rs:69` `clear-openai-key` | Bare `?` | Succeeds vacuously with a note. |

`keychain::exists` needs no change: it funnels through `get` and has an `Err(e) => Err(e)` catch-all, and its only consumers already treat a hard error as fatal.

The distinction that matters: `get` / `set` / `delete` keep returning the typed `NoStore` error, because that is what lets the CLI commands give guidance. Only `resolve_openai_key_with`, whose whole contract is "configured or not", collapses it to `Ok(None)`.

See 4.2.

### 3.3 macOS CLI binary ships unsigned

The macOS CLI archive is not Developer ID signed and not notarized. Rationale:

- **Homebrew formulae are not quarantined.** Homebrew does not set `com.apple.quarantine` on formula-installed binaries, which is why `--no-quarantine` has only ever applied to casks. `brew install` is the recommended macOS path, and it runs an unsigned binary cleanly. Signing buys nothing there.
- Signing **without** notarizing would not clear quarantine on a direct download either, so it is ceremony without an outcome.
- Notarizing a bare Mach-O gives online-only verification (no stapling) in exchange for a second `notarytool` round-trip per release, to serve the narrow set of users who download the tarball from the Releases page on the one OS where the tap works best.
- This matches the Rust CLI ecosystem norm (ripgrep, fd, bat, starship).

"Unsigned" here means *not Developer ID signed*. Rust ad-hoc signs `aarch64-apple-darwin` binaries at link time, because Apple Silicon refuses to execute an entirely unsigned Mach-O.

The README documents `xattr -d com.apple.quarantine` for direct downloads and points macOS users at `brew install` first.

The **desktop cask** is a different case and is fine: the DMG is signed and notarized as of v0.5.0. Homebrew is deprecating `--no-quarantine` and removing casks that fail Gatekeeper checks from 1 September 2026, so a cask is only viable at all because that work already happened.

### 3.4 The tap updates on `release: published`, not inside `release.yml`

`release.yml` creates a **draft** release (`releaseDraft: true`). Draft asset URLs do not resolve for anonymous users, and Homebrew users are anonymous - so a formula written during the build run would point at a URL that 404s until the maintainer clicks Publish.

The tap update is therefore a separate workflow triggered `on: release: types: [published]`. This is the correct trigger independent of the draft question: it fires exactly when the asset URLs become publicly resolvable.

**It also needs a `workflow_dispatch` trigger taking an explicit release tag.** A `release` event runs the workflow file **as it exists at the release's tag ref**, not as it exists on `main`. So re-running a failed run replays the same broken file, and a fix merged to `main` is never picked up - there is no second `published` event for an already-published release. Without a dispatch path, a bug found on the first real run would be unrecoverable short of cutting another release.

The dispatch trigger buys three things: replay after a fix, a pre-publication validation route against an older tag, and recovery if the tap push fails for an unrelated reason (an expired token, a tap-repo conflict). The update must therefore be idempotent - re-running against a tag whose formula is already current is a no-op, not an error. This is what makes 3.4's "the formula can be re-pushed without rebuilding anything" actually true; on the `release` trigger alone it was not.

### 3.5 The draft release stays

Considered and rejected: having `release.yml` publish directly to simplify sequencing.

The header comment's stated reason (notarization needs eyeballing, and the per-OS validation matrix is not covered by the per-PR gate) still holds. The stronger reason is that **the matrix legs race**: `tauri-action` creates the release as soon as the first leg finishes, with `fail-fast: false`. Publishing directly would leave a multi-minute window where a public release carries macOS assets but no Windows ones - and v0.5.1 widens that window, because the `cli` job runs `needs: build`, so four more archives land after the desktop ones. The release notification and the "Latest" pointer would both fire on a partial state, and un-publishing does not recall a notification.

The draft makes every partial state invisible. Per 3.4, it costs the tap nothing.

### 3.6 Release notes live in one file

`release.yml` duplicates the release body verbatim across both legs (lines 147-151 and 164-168), because a step's `with:` block cannot be shared conditionally. They are byte-identical today only by hand.

This release adds CLI asset lines and brew instructions to that body, which would mean editing the same prose in two places twice. Instead: the body moves to `.github/release-body.md`, and a small `notes` job reads it and exposes it as a job output that both legs consume via `releaseBody: ${{ needs.notes.outputs.body }}`.

Rejected as too large for this tag: a `prepare` job that creates the draft up front with `gh release create --draft --notes-file`, leaving every build leg to upload into an existing release. Conceptually cleaner and it would remove the leg-creates-the-release race entirely, but it reworks a path that only just proved itself end-to-end at v0.5.0.

### 3.7 New CI secret

The tap workflow pushes to a different repository, which `GITHUB_TOKEN` cannot do. It needs a fine-grained PAT scoped to `contents: write` on `Oszkar/homebrew-balanze` only, stored as `TAP_PUSH_TOKEN`.

This is the AGENTS.md §8 "expanding the secrets surface" item, accepted by putting the tap in scope. It follows the precedent set by the Apple signing secrets: documented in a workflow comment at the point of use, not added to AGENTS.md §3.4, which scopes *user* secrets (OpenAI keys, Claude OAuth) rather than CI credentials.

### 3.8 `deny.toml` and the three-list rule

AGENTS.md §1 requires `docs/PRD.md`'s platform matrix, `release.yml`'s build matrix, and `deny.toml`'s `[graph].targets` to agree exactly. Adding two triples means touching all three.

**This will likely break `cargo-deny`, which is a required status check on `main`.** `deny.toml`'s own comment says why: *"Tauri pulls GTK3 transitively on Linux, which is RustSec-unmaintained, but we don't ship that path so we shouldn't fail CI on it."* Adding `x86_64-unknown-linux-musl` pulls `src-tauri`'s Linux dependency path into the vetted graph for the first time.

Resolution: add both triples, then add whichever gtk-rs advisories fire to `[advisories].ignore`, each with a comment stating that `src-tauri` is never built for Linux and the advisory cannot reach a user. The invariant survives with a documented exception rather than a silent one.

The rule itself also needs restating, because `release.yml` now has **two** matrices: `deny.toml`'s targets are the **union** of the desktop and CLI matrices. AGENTS.md §1 and the `deny.toml` comment both need that wording.

### 3.9 Tap layout

Repository `Oszkar/homebrew-balanze`, created manually before PR 3:

- `Formula/balanze-cli.rb` - a binary formula with `on_macos` / `on_arm` and `on_linux` / `on_intel` blocks pointing at the release archives, `MIT` license, `def install; bin.install "balanze-cli"; end`, and a `test do` block asserting `balanze-cli --version` reports the formula version.
- `Casks/balanze.rb` - the notarized `Balanze_<version>_aarch64.dmg`.

The tap workflow reads checksums the release already advertises rather than re-hashing, so the tap can only ever agree with the release. **The two artifact families carry their checksums differently, and the workflow must handle both:**

- **CLI archives** (the formula) have sibling `.sha256` files, added by this milestone. One `cut -d' ' -f1` per file.
- **The DMG** (the cask) has no sibling. It is covered by `macos-aarch64-checksums.txt`, written by `release.yml`'s existing checksum step (`release.yml:203`), which globs the bundle directory and emits `<hash>  <filename>` lines. The cask's hash comes from `awk`-ing that file for `Balanze_<version>_aarch64.dmg`.

Adding a DMG sibling in the CLI job was considered and rejected: the desktop legs already publish a checksum file, and duplicating one hash in two formats invites them to disagree. The tap workflow fails loudly if either lookup comes back empty, rather than writing an empty `sha256` that Homebrew would reject at install time with a confusing error.

## 5. Validation

### 5.1 Gates

PR 0 is documentation only. PR 1 is covered by the standard matrix: `cargo build --workspace`, `cargo clippy --workspace --all-targets -- -D warnings`, `cargo nextest run --workspace`, `cargo fmt --all -- --check`. PR 2 additionally needs `cargo deny check` to pass with the new targets - that is the gate most likely to fail first.

### 5.2 The part CI cannot cover

`release.yml` is `workflow_dispatch`-only and its tag regex (`^v[0-9]+\.[0-9]+\.[0-9]+$`, line 47) rejects `-rc` suffixes, so **PR 2's workflow changes cannot be validated by any per-PR gate.**

Pre-merge dry run: push a throwaway `v9.9.9` tag from the feature branch. It satisfies the regex, exercises all four CLI legs plus both desktop legs on real runners, and produces a draft. Inspect the assets, then delete the draft release and the tag. Nothing public is touched and no real version is consumed.

The dry run necessarily re-runs the macOS signing and notarization path and consumes one notarization submission. That is a feature rather than waste: PR 2 changes the desktop legs too (both switch to the shared `notes` output per 3.6), so those legs need exercising anyway.

### 5.3 Manual tests per boundary

- **PR 1**: on Linux, `balanze-cli setup` and `balanze-cli set-openai-key` surface the guidance rather than a keyring error; `balanze-cli doctor` reports the keychain line as a warning, not a failure; and `balanze-cli status` shows the OpenAI cell as not-configured rather than errored. On Windows and macOS, `doctor` output is unchanged.
- **PR 1, required by AGENTS.md §6**: this touches `crates/keychain`, so the `#[ignore]`'d real-keychain roundtrip must be run by hand on **both** Windows and macOS - `cargo nextest run -p keychain --run-ignored all`. CI keychain access is unreliable, so a green PR does not certify it. Run it at PR time rather than only before tagging, so a regression in the store-registration path is caught while the change is fresh.
- **PR 2**: the `v9.9.9` dry run; then extract each archive and run `balanze-cli status` on Windows x64, macOS arm64, and a Linux host. **The arm64 Windows binary has no test host** - it ships build-verified only, which is the same "ships untested" trade-off #219 flags for the desktop.
- **PR 3**: against a real published release, on **both** OSes Homebrew serves:
  - macOS: `brew install oszkar/balanze/balanze-cli` (expect no Gatekeeper prompt - formulae are not quarantined) and `brew install --cask oszkar/balanze/balanze`.
  - **Linux (Linuxbrew, x64):** `brew install oszkar/balanze/balanze-cli`. This is a stated done condition and the formula's `on_linux` / `on_intel` block is a completely separate code path from the macOS one - a typo there is invisible to a macOS-only test. Untap first so the install is clean.
  - Verify the `workflow_dispatch` replay path by dispatching `tap.yml` a second time against the same tag and confirming it is a no-op rather than an error.

## 7. Risks

| Risk | Mitigation |
|---|---|
| `cargo-deny` fails on the Linux GTK3 graph, blocking a required check | Expected, not a surprise - 3.8 plans the ignore entries. If the ignore list grows unreasonably, fall back to excluding `src-tauri` from the Linux graph and record why. |
| `windows-latest` lacks the ARM64 MSVC build tools | Anticipated by the conditional done condition in section 2, not a mid-flight improvisation. Drop the leg from `release.yml`, `deny.toml`, the PRD matrix, and the README table **in one commit** so the three lists never disagree, and update #219. Does not block the milestone. Do not install ARM64 toolchains in CI to force it through. |
| The `v9.9.9` dry run leaves a stray tag or draft | Delete both explicitly as the last step of the dry run; both are on the feature branch and never reach `main`. |
| The tap formula drifts from the release | The tap workflow derives version, URL, and sha256 from the published release itself, so drift requires the workflow to fail rather than to lie. A failed tap run is visible in Actions. |
| `TAP_PUSH_TOKEN` leaks or expires | Fine-grained, single-repo, `contents: write` only. A leak grants formula edits on a personal tap and nothing else. Rotation is a token regeneration. |

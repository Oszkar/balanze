# v0.5.0 - Distribution & Legibility (first slice) - Milestone Spec

Date: 2026-07-16
Status: approved (scope), ready for implementation plan
Phase: 5 (Distribution & Legibility), first sub-milestone

## 1. Context

Phase 5 in `docs/PRD.md` ("Distribution & Legibility") bundles a large set of deliverables: a runnable desktop release, CLI distribution, README/how-it-works legibility, a price-table refresh script, a "Send Logs" menu item, and optional signing/store work. That is too much for one tag.

Phase 4 shipped as sub-milestones under patch bumps (`v0.4.0` UI -> `v0.4.1` CLI -> `v0.4.2` statusline -> `v0.4.3` Codex -> `v0.4.4` hardening). Phase 5 follows the same cadence. This spec defines **v0.5.0**, the first bounded slice.

The macOS sign + notarize path is already wired into `.github/workflows/release.yml` (PR #201). No release has ever attached binaries (`v0.4.4`'s assets are empty), so v0.5.0 is the first binary release and the first real end-to-end run of the notarization path.

## 2. Scope

v0.5.0 theme: **a downloadable, discoverable, credible desktop release.** First binary release; the tray app behaves like a real tray app (starts at login); the repo reads as a serious project.

### Deferred to v0.5.1+

### Done condition

## 3. Settled decisions

### 3.1 Autostart - approach A (plugin-only, OS is source of truth)

Use the official `tauri-plugin-autostart`. The OS login-item state is the single source of truth via `enable()` / `disable()` / `is_enabled()`. **No `Settings` schema field**, nothing to reconcile on startup, and the toggle stays correct if the user clears the login item through the OS. Rejected approach B (a `Settings.launch_at_login` field reconciled on startup) because it adds state that can drift from the OS.

### 3.2 Windows installer - unsigned

Ship unsigned MSI/NSIS. Matches the release notes already committed in `release.yml` ("SmartScreen will warn on first run; click More info -> Run anyway") and the PRD's "Windows code-signing remains optional, not load-bearing." macOS is signed/notarized via #201.

### 3.3 Docs site - adopt mdBook, build in v0.5.1

Record the decision to use mdBook (Rust-native, lowest new-tooling surface, the #158 leading candidate). v0.5.0 ships only the recorded decision; the skeleton + Pages workflow + content land in v0.5.1 with the how-it-works writeup. This satisfies #158's acceptance ("a decision recorded").

### 3.4 CLI distribution via crates.io - out by design

Verified: the workspace sets `publish = false` with an explicit "never published to crates.io" comment, and `balanze_cli` depends on 15 internal path crates with generic names (`window`, `keychain`, `logging`, `settings`, `backoff`, ...). Publishing to crates.io would mean publishing all 15 into a shared namespace. So CLI distribution (deferred to v0.5.1) will be **prebuilt CLI binaries + a Homebrew tap**, not `cargo install` from crates.io. Recorded here so v0.5.1 planning starts from the right premise.

## 4. PR 1 - `feat(app): launch at login (#106)`

Adds the autostart feature only. The Phase-5 roadmap documentation (§2 item 5) lands ahead of this on the docs branch, so PR 1 stays a focused feature PR that touches no docs beyond code-adjacent comments.

### 4.1 Dependency

- Add `tauri-plugin-autostart` to `src-tauri/Cargo.toml`, pinned at the workspace level alongside the other `tauri-plugin-*` crates on the same minor. This is the §8 "new dependency" item, accepted by putting #106 in scope.
- No `@tauri-apps/plugin-autostart` JS package - the toggle calls Rust app commands, so no new webview capability is needed (PoLP; see 4.3).

### 4.2 Rust wiring

- **Plugin registration (`src-tauri/src/lib.rs`).** `single_instance` is registered first at `lib.rs:607` and **must stay first** (CLAUDE.md known issue); add `tauri_plugin_autostart::init(...)` immediately after the single-instance block, before `tauri_plugin_opener`. Configure the plugin for the app's launch flag set (start hidden in the tray, consistent with the current hidden-on-launch behavior).
- **Commands (`src-tauri/src/commands.rs`).** Two app commands live alongside the existing `hide_window` / `resize_popover` (same file, same pattern):
  - `get_launch_at_login() -> Result<bool, String>` - returns `is_enabled()`.
  - `set_launch_at_login(enabled: bool) -> Result<(), String>` - calls `enable()` / `disable()`.
- Errors follow §3.2 of AGENTS.md: `Result<_, String>` derived from the underlying error's `to_string()`.
- Register both commands in the `tauri::generate_handler!` list in the `invoke_handler` at `lib.rs:617` (next to `commands::hide_window`, `commands::resize_popover`).

### 4.3 Capability

No change to `capabilities/default.json`. The toggle invokes the two Rust app commands (own `#[tauri::command]`s need no capability entry; capabilities gate only core/plugin APIs). This is deliberate - calling `@tauri-apps/plugin-autostart` from JS would instead require an `autostart:` capability.

### 4.4 Frontend

- `src/lib/ipc.ts` - add `getLaunchAtLogin()` / `setLaunchAtLogin(enabled)` wrappers over the two commands.
- `src/lib/components/SettingsView.svelte` - add a "Launch at login" toggle in the existing settings column (natural home alongside the provider toggles). On mount, read `getLaunchAtLogin()` to set initial state; on change, call `setLaunchAtLogin(next)`. Use Svelte 5 runes (`$state`), matching the surrounding component.
- Gallery: if the settings panel appears in `src/lib/gallery/`, the toggle should render there too (IPC stubbed to a no-op per the gallery contract).

### 4.5 Roadmap documentation (landed ahead on the docs branch, not in PR 1)

The `docs/PRD.md` Phase-5 decomposition ships on this spec's docs branch and merges before implementation. Recorded here for traceability; the edit is already made. Its content:

- Phase 5 split into **v0.5.0** (runnable desktop release + launch-at-login + README legibility + docs-site decision) and **v0.5.1+** (CLI distribution with the crates.io-out-by-design note, how-it-works writeup, mdBook skeleton + Pages build, price-table refresh script, Send Logs).
- The launch-at-login double-listing removed from the v0.7 "Refresh cadence & startup" bullet (autostart now ships in v0.5.0).

PR 1 therefore edits no PRD content.

### 4.6 Manual tests (PR 1 checkpoint)

- `bun run tauri dev` smoke: settings panel opens, toggle renders and reflects current OS state.
- **Windows 11:** toggle on -> confirm HKCU `...\Windows\CurrentVersion\Run` has a Balanze entry -> reboot -> Balanze appears in the tray. Toggle off -> Run entry gone -> reboot -> does not appear.
- **macOS 15+:** toggle on -> confirm a LaunchAgent is registered -> reboot/re-login -> Balanze appears in the menu bar. Toggle off -> reboot -> does not appear.
- Toggle reflects OS state on next open after an external change (clear the login item via the OS, reopen settings, toggle shows off).

### 6.2 Maintainer runbook (human-gated)

1. Confirm `cargo build --workspace` + clippy + nextest + `bun run check` pass locally (src-tauri clippy and platform builds only run on schedule/dispatch per the AGENTS.md CI split, so run them locally before tagging).
2. Merge PR 3, then tag `v0.5.0` on the merge commit and push the tag.
3. Trigger `release.yml` via `workflow_dispatch` with `tag=v0.5.0`.
4. **Verify macOS:** download the DMG/app on the MacBook, confirm it opens without a Gatekeeper warning (signed + notarized), tray icon appears, popover opens, Quit exits cleanly.
5. **Verify Windows 11:** download the MSI/NSIS, install, confirm the tray app runs (accept the SmartScreen "More info -> Run anyway"), popover opens, Quit exits cleanly.
6. Publish the draft release with assets attached.
7. If notarization fails: diagnose (Apple secrets are already configured), fix, re-run. Success is the v0.5.0 done condition.

## 8. Effort

~2-3 focused dev-days plus manual verification on both OSes.

- PR 1: S-M (dep + two commands + toggle + PRD edit; per-OS reboot verification is the time cost).
- PR 2: S (screenshot capture + README edit + issue update).
- PR 3: S code (version bump + CHANGELOG); M risk/manual (first notarized run + two-OS install verification).

## 9. Risks

1. **First-ever notarized release.** `release.yml` + #201 wired it but it has never run to publish assets; the first real run may surface Apple notary/keychain issues. Highest-variance item; explicitly inside the v0.5.0 done condition (fix if it fails).
2. **Two-OS manual verification** is unavoidable (login items and installers can't be CI-tested). MacBook + this Windows 11 machine cover both.
3. **Plugin registration order** - autostart after single_instance; a wrong order risks the double-launch issue the single-instance plugin guards against.
4. **Roadmap doc lands first** - the PRD Phase-5 decomposition merges on the docs branch before PR 1 / PR 2, so neither feature PR edits the PRD and there is no cross-PR PRD conflict.
